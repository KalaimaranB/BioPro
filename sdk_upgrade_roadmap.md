## 🧠 Understanding the Trust System (Plain English)

Before working on any of this, here is a clear mental model of how the whole thing works end-to-end:

```
BioPro Core Authority (Root)
        │
        │  signs →
        ▼
Developer Certificate (dev_cert.bin)
  [32 bytes: developer's public key]
  [64 bytes: root's signature of that key]
        │
        │  developer uses private key to →
        ▼
Plugin Manifest Signature (signature.bin)
  [64 bytes: developer's signature of manifest.json contents]
        │
        │  TrustManager verifies all three layers at boot →
        ▼
Plugin is Loaded ✅ (or rejected ❌)
```

**What `trust_chain.json` is:** A linked list of "who certified who," from the developer up to the root. Allows intermediate authorities (like a university) to sign developer keys, so BioPro Core doesn't have to sign every developer individually.

**What `authorities.json` is:** A remote ledger of intermediate Certificate Authorities (e.g., UBC). The app fetches this at boot and saves their public keys to `~/.biopro/trusted_roots/`. This means you can add a new university as a trusted authority without releasing a new app version.

**What `registry.json` is:** A catalog of available plugins. Contains download URLs, version numbers, minimum app version requirements, and which authority/developer signed each one. Has nothing to do with trust verification itself — that's entirely handled by the files inside the plugin `.zip`.

**What `trust_overrides.json` is:** A local escape hatch. If you are a developer actively editing a plugin locally, its hash will never match the signed manifest. You can "lock" the current state locally — this creates a machine-specific snapshot. This file is signed by a local per-machine key so it can't be tampered with from outside.

## 🩹 Phase 3: Patchability Architecture — What Can and Cannot Be Hot-Patched

This is critical to understand. BioPro uses **User-Space Delta Patching** to avoid breaking macOS Gatekeeper's app bundle signature.

### ✅ What CAN Be Hot-Patched (User Space)

These files live entirely in `~/.biopro/` and can be updated by `NetworkUpdater` at any time without a full app release:

| Asset Type | Location | Notes |
|:---|:---|:---|
| Documentation pages | `~/.biopro/docs/*.md` | Fetch individual changed files from `biopro-distribution` |
| Plugin `.zip` files | `~/.biopro/plugins/<id>/` | Downloaded and verified by `PluginInstallerWorker` |
| `registry.json` cache | `~/.biopro/plugins/installed.json` | Auto-synced from remote on boot |
| Trusted developer keys | `~/.biopro/trusted_roots/*.pub` | Synced by `_sync_keys()` from `authorities.json` |
| AI model files | `~/.biopro/models/` | Downloaded on-demand |

### ❌ What CANNOT Be Hot-Patched (Requires Full App Release)

These files are baked into the signed `BioPro.app` bundle:

| Asset Type | Why it can't be patched |
|:---|:---|
| `biopro/core/*.py` — Core logic | Modifying the app bundle breaks the macOS Gatekeeper signature |
| `biopro/ui/*.py` — UI windows | Same — sealed inside the app bundle |
| `biopro/sdk/*.py` — SDK base classes | Same |
| `_FALLBACK_ROOT_PUBLIC_KEY_HEX` | Hardcoded at compile time; changing it requires a new build |
| Python interpreter & PyQt6 | Bundled by PyInstaller inside the app |
| Themes (`.json` files in `biopro/themes/`) | Currently bundled — *consider* moving to `~/.biopro/themes/` for hot-swap |

> [!NOTE]
> **The Grey Zone — Themes:** Themes are currently bundled inside the app (cannot be hot-patched). Consider refactoring the `ThemeEngine` to first check `~/.biopro/themes/` before falling back to built-in themes. This would allow new community themes to be delivered as user-space patches.

### Implement the Delta-Patching Manifest

The `NetworkUpdater` currently only fetches `registry.json` for plugin state. It needs a separate `patch_manifest.json` that describes user-space assets:

- [ ] **Create `biopro-distribution/patch_manifest.json`**:

      "07_Module_Author_Guide.md": { "sha256": "def...", "url": "..." }
    },

      "tri_state_innovation.json": { "sha256": "ghi...", "url": "..." }

## 🤝 Phase 8: Collaborative & Multi-Author Scaling (SOLID & TDD Approach)

Transitioning from solo-developer plugins to team-based projects requires a systematic architectural shift in the cryptographic trust system, guided strictly by SOLID principles and TDD.

### 1. Refactoring the Trust Architecture (SOLID Focus)
- [ ] **Entity-Based Certificates**: Abstract the rigid `Developer` identity into a flexible `Entity` (Project, Team, or Person). Introduce an `ITrustStrategy` interface to adhere to the Open/Closed Principle.
- [ ] **TDD Strategy for TrustManager**: Write red-tests for `ProjectTrustStrategy` (e.g., `test_project_trust_chain_validates_successfully()`) before splitting the monolithic `TrustManager` into `CertificateReader`, `ChainValidator`, and `SignatureVerifier` (Single Responsibility Principle).
